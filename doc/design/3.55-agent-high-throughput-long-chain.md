# 3.55 AI Agent 高吞吐长链路增强

## 问题背景

RedC Agent 在 10 轮以内的短链路任务表现良好，但 15+ 轮的长链路任务（如：多节点部署 + 配置 + 验证）不稳定。根因分析发现 5 个系统性瓶颈：

| 瓶颈 | 现状 | 影响 |
|------|------|------|
| 无 token 计数 | `aiMessages` 无限增长 | 15+ 轮 context 溢出 |
| max_tokens 硬编码 4096 | 工具 JSON 被截断 | 多工具调用失败 |
| 无中间状态持久化 | 中断即丢失 | 长任务重头再来 |
| 串行工具执行 | 逐个执行 | 多工具耗时翻倍 |
| 错误无分级 | 所有失败消耗一轮 | 瞬时错误浪费轮次 |

## 实现方案

### Enhancement 1: 上下文窗口智能管理

**文件**: `mod/ai/client.go`, `app_ai_chat.go`, `mod/profile.go`

#### Token 估算

```go
func EstimateTokens(messages []Message) int
func estimateContentTokens(s string) int
```

- 中文字符：~0.67 tokens (2/3)
- ASCII 字符：~0.25 tokens (1/4)
- 每条消息额外 4 tokens overhead

#### Context Budget

```go
contextBudget := 108000 // 默认 ~90% of 120K
if aiConfig.ContextWindow > 0 {
    contextBudget = aiConfig.ContextWindow * 9 / 10
}
```

用户可通过 `AIConfig.ContextWindow` 配置模型实际上下文窗口大小。

#### 智能压缩 `compactMessages`

触发条件：`EstimateTokens(aiMessages) > contextBudget`

压缩策略：
- **保留完整**：system prompt + 所有 user 消息 + 最近 3 轮交互（~6条 assistant/tool 消息）
- **压缩旧消息**：
  - assistant 消息 → 前 200 字符 + `[called: tool1, tool2]`
  - tool 结果（成功）→ 前 100 字符
  - tool 结果（失败）→ 前 300 字符（保留更多错误上下文）

轻量压缩 `compressOldToolResults` 在 context 未超限但已执行 2+ 轮时触发。

### Enhancement 2: max_tokens 动态化

**文件**: `mod/ai/client.go`

Anthropic 路径 `max_tokens` 从 4096 提升至 8192，确保工具调用模式下有足够输出空间。

### Enhancement 3: 断点持久化与恢复

**文件**: `mod/memory.go`, `app_ai_chat.go`, `AIChat.svelte`

#### 数据库扩展

```sql
ALTER TABLE agent_tasks ADD COLUMN checkpoint_messages TEXT;
ALTER TABLE agent_tasks ADD COLUMN checkpoint_round INTEGER DEFAULT 0;
ALTER TABLE agent_tasks ADD COLUMN prompt_template TEXT;
```

#### 保存机制

Agent loop 每 5 轮自动保存完整 `aiMessages` JSON 到 `checkpoint_messages`：

```go
if round > 0 && round%5 == 0 {
    cpJSON, _ := json.Marshal(aiMessages)
    memoryStore.SaveCheckpoint(conversationId, string(cpJSON), round, promptTemplate)
}
```

#### 恢复机制

`ResumeAgentStream(conversationId)` 方法：
1. 从 `agent_tasks` 表加载 checkpoint_messages
2. 反序列化为 `[]ai.Message`
3. 提取 user/assistant 消息重建 `AIChatMessage` 列表
4. 重新调用 `runAgentLoop` 继续执行

#### 前端

- `lastInterruptedConvId` 状态追踪中断的对话 ID
- 超时（`⏱️`）和错误中断后显示琥珀色恢复栏
- "恢复执行" 按钮调用 `ResumeAgentStream`

### Enhancement 4: 并行工具执行

**文件**: `app_ai_chat.go`

#### 判断逻辑

`canParallelizeToolCalls(calls)` 返回 false 的工具（必须串行）：
- 交互工具：`ask_user`, `update_plan`
- 写操作：`start_case`, `stop_case`, `kill_case`, `compose_up`, `compose_down`, `exec_command` 等
- 文件操作：`save_template_files`, `save_compose_file`, `delete_template` 等

其余只读工具（`get_case_outputs`, `list_cases`, `get_ssh_info` 等）可并行。

#### 执行流程

```
if len(toolCalls) > 1 && canParallelizeToolCalls(toolCalls) {
    // goroutine per tool call + WaitGroup
} else {
    // 串行执行（向后兼容）
}
```

### Enhancement 5: 工具错误自动重试

**文件**: `app_ai_chat.go`

#### 可重试错误

`isRetryableError(err)` 匹配以下模式：
- `connection refused` / `connection reset` / `i/o timeout`
- `broken pipe` / `no route to host` / `dial tcp`
- `failed to create SSH client` / `failed to create SSH session`

#### 重试策略

在 `executeSingleTool` 中：
```go
if execErr != nil && isRetryableError(execErr) {
    time.Sleep(5 * time.Second)
    result, execErr = mcpServer.ExecuteTool(name, args) // 重试 1 次
}
```

透明重试，不消耗 Agent 思考轮次。
